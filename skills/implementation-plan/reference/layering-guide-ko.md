# 레이어링 가이드 (참조)

## 4개 레이어
| 레이어 | 책임 | 예 |
|---|---|---|
| 표현(Presentation) | 입출력, UI, HTTP 라우팅 | React 컴포넌트, API 라우터 |
| 애플리케이션(Application) | 유스케이스 조율, 트랜잭션 | 서비스/핸들러 |
| 도메인(Domain) | 핵심 규칙·엔티티 | 도메인 모델, 검증 규칙 |
| 데이터(Data) | 영속화, 외부연동 | 리포지토리, ORM, API 클라이언트 |

## 규칙
- **의존 방향은 안쪽으로**: 표현→애플리케이션→도메인. 도메인은 프레임워크/DB를 모른다.
- SQL·ORM은 데이터 레이어에만. 비즈니스 규칙은 도메인/애플리케이션에.
- 계약(인터페이스)으로 레이어를 분리 → 테스트 시 목(mock) 주입 용이.

## 규모별 실용 지침
- **소규모**: 레이어를 과하게 쪼개지 말 것(표현+애플리케이션 합침 허용). 폴더로만 구분.
- **중대규모**: 명확한 모듈 경계, 도메인 분리, 의존성 주입.

## 프론트/백엔드 병렬 개발
1. `api-spec.md`(OpenAPI)로 계약 확정.
2. 백엔드는 계약대로 구현, 프론트는 목 서버로 병렬 진행.
3. 통합 시 계약 준수 검증(계약 테스트).

## 테스트 용이성(4단계 대비)
- 순수 로직(도메인)은 부수효과 없이 → 단위테스트 쉬움.
- 외부연동은 경계 뒤로 → e2e/통합테스트 대상 명확.

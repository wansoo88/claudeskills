# 테스트 가이드 (참조) — 무료 도구 기준

## 테스트 피라미드
| 층 | 비중 | 도구(무료) | 특징 |
|---|---|---|---|
| 단위 | 많이 | Pytest(Py), Jest/Vitest(JS/TS) | 빠름·결정적, 로직/경계 |
| 통합 | 적당 | pytest + testcontainers, docker compose | DB·연동 경계 |
| e2e | 핵심만 | **Playwright**(대안 Cypress) | 사용자 시나리오 |

## 좋은 테스트 원칙
- **결정적**: 시간·랜덤·네트워크 의존 제거(고정 시드·목·가짜시계).
- **독립적**: 순서 무관, 격리된 데이터. 테스트 간 상태 공유 금지.
- **의미 있는 assert**: 동작(behavior) 검증, 구현 세부 아닌 계약.
- **실패 케이스도**: 예외·권한거부·유효성 실패 경로.

## 단위 (예: Pytest)
- AAA 패턴(Arrange-Act-Assert). 파라미터라이즈로 경계값.
- 순수 함수는 목 없이. 외부 의존은 경계에서 목.

## e2e (Playwright)
- 핵심 유저 여정만(로그인→핵심작업→검증). 과하게 늘리지 말 것.
- 셀렉터는 역할/텍스트 기반(견고). 자동 대기 활용, 임의 sleep 금지.
- CI에서 헤드리스 실행, 실패 시 스크린샷/트레이스 저장.

## 커버리지
- 숫자 목표보다 **핵심 로직·리스크 높은 경로** 우선.
- 커버리지는 안전망이지 품질 증명이 아님(의미 있는 assert가 핵심).

## 성공기준(KPI) 연계
- requirements.md의 성공기준을 **검증 가능한 테스트**로 1:1 매핑.
- 예: "리포트 시간 90% 절감" → 자동화 경로 e2e + 처리시간 측정.

## CI 통합
- PR마다 단위+통합 자동 실행, e2e는 핵심 세트. GitHub Actions/사내 CI.

# README 작성 (프로젝트 산출물)

목표: 이 프로젝트를 **처음 보는 사람이 빠르게 이해하고 직접 실행**할 수 있는 README.md를 만든다. 인수인계(6단계)의 핵심 산출물이자, 언제든 갱신 가능한 유틸리티.

> 왜 스크립트가 아니라 스킬인가: README는 프로젝트 맥락을 읽고 **판단해 쓰는 작업**이라 템플릿 자동채움보다 Claude가 직접 작성하는 게 품질이 높다. 구조/원칙은 아래 reference를 따른다.

## 진행 순서

### 1. 소스 읽기
- `CLAUDE.md`(스택·역할·현황), `skill.md`(진행 단계), `docs/`(요구사항·설계·테스트·모니터링).
- 실제 코드/설정: `package.json`·`requirements.txt`·`Dockerfile`·`docker-compose.yml`·`.env.example` 등에서 **실행법**을 확인(추측 금지, 없으면 표시).

### 2. 구조 잡기
`reference/readme-template-ko.md`의 구조·작성 원칙을 따른다. 프로젝트 성격(대시보드/API/파이프라인 등)에 맞게 섹션을 가감.

### 3. 작성 (`README.md`)
- 한 줄 설명 + 개요(무엇을·누가·왜).
- **빠른 시작**: 사전요구 → 설치 → 실행(가능하면 `docker compose up` 한 줄). 복붙 가능한 명령으로.
- 기술 스택, 프로젝트 구조(폴더 지도), 환경변수/설정, 테스트 실행법, 배포/운영(대시보드·알람), 문의.
- 기존 README가 있으면 **덮어쓰기 전 확인**하고, 좋은 부분은 살려 병합.

### 4. 검증
- 적은 명령이 실제로 존재하는지(스크립트/파일) 교차 확인.
- "처음 보는 사람이 이대로 따라 하면 실행되는가?" 자문.

### 5. 마무리
무엇을 만들었는지 한국어로 요약. Confluence 등 외부 게시는 사용자 확인 후.

## 하지 말 것
- 실제로 없는 명령·경로를 지어내기. 장황한 마케팅 문구. 최신 상태와 어긋난 정보 방치.

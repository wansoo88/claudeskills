# AI 개발팀 스킬셋 — 전체 설계도 (v0.1 초안)

> 목적: 데이터팀 팀장(당신)이 **부족한 역할(웹/WAS 설계·개발·QA)을 AI로 채워** 데이터 프로덕트를 1인 개발하고, 그 역량을 팀에 전파하기 위한 **재사용 Claude 스킬셋**의 청사진.
> 근거: `research/claude-skills-트렌드-리서치-2026-07.md` (검증 25/25).
> ⚠️ 이 문서는 **합의용 초안**입니다. 착수 전에 §8의 결정사항을 확정합니다.

---

## 1. 큰 그림 — 4개 부품이 어떻게 맞물리나

리서치 결론: 필요한 인프라는 Anthropic에 이미 다 있음. 우리는 **부품을 조립**만 하면 됩니다.

| 부품 | 위치 | 역할 | 우리 용도 |
|---|---|---|---|
| **Subagent** | `agents/*.md` | 격리된 전문 역할 | 7개 시니어 역할 + 2개 리뷰어 |
| **Skill** | `skills/*/SKILL.md` | 절차·지식·산출물 생성법 | 6단계 파이프라인 각 절차 |
| **Hook** | `hooks/` | 결정론적 강제 게이트 | code/security-review "무조건" 강제 |
| **Plugin** | 전체를 한 폴더로 | 패키징·배포 단위 | 팀 공유 + 월간 자동갱신 |

> 조립 원칙(검증됨): **역할=subagent, 절차=skill, 강제=hook, 배포=plugin, 외부연결=MCP.** 스펙 추적은 `agentskills.io`.

## 3. 9개 역할(Subagent) 정의

| # | 역할 | 담당 단계 | 핵심 산출물 | 도구 권한(원칙: 최소) |
|---|---|---|---|---|
| 1 | **service-strategist** 서비스 전략가 | 1 인터뷰 | 요구사항·서비스정의·성공지표(KPI) | AskUserQuestion 중심, 읽기 |
| 2 | **infra-architect** 인프라 아키텍트 | 2·5 | AWS 아키텍처(draw.io)·네트워크·모니터링 | 설계 도구, MCP |
| 3 | **software-architect** SW 아키텍트 | 2·3 | 레이어 설계·구현계획서 | 읽기·쓰기 |
| 4 | **db-architect** DB 아키텍트 | 2 | ERD/UML·스키마·인덱스 | 읽기·쓰기 (당신 강점→검수자) |
| 5 | **frontend-react** 프론트 | 3 | React 반응형 컴포넌트 | 프론트 파일 쓰기 |
| 6 | **backend-docker** 백엔드 | 3 | Docker 기반 API·서비스 | 백엔드 파일·bash |
| 7 | **qa-engineer** QA | 4 | 단위·e2e 테스트·리포트 | 테스트 파일·실행 |
| G1 | **code-reviewer** (게이트) | 3 | 코드리뷰 리포트 | 읽기 전용 |
| G2 | **security-reviewer** (게이트) | 3 | 보안리뷰 리포트 | 읽기 전용 |

> 설계 근거: 각 subagent는 **독립 컨텍스트**로 격리 → 역할 간 오염 없음. `description`을 상세히 써서 위임 정확도↑. **자동위임은 불안정하므로 파이프라인에서 이름으로 명시 호출.**

## 4. 6단계 파이프라인 (단계 → 산출물 → 담당 → 게이트)

당신이 정의한 단계·산출물을 그대로 반영:

| 단계 | 산출물 | 담당 | 필수 게이트 |
|---|---|---|---|
| **1. 인터뷰** | `requirements.md` + `CLAUDE.md`·`skill.md` 자동생성 | service-strategist | **사용자 승인**(AskUserQuestion 한국어) |
| **2. 설계** | AWS 아키텍처(draw.io)·디자인·ERD/UML·보안설계 | infra/sw/db-architect | 설계 리뷰 |
| **3. 구현** | 구현계획서 + 레이어별 코드 | sw-architect → frontend/backend | **code-review 무조건 + security-review 무조건** (hook 강제) |
| **4. 테스트** | 단위테스트 + e2e테스트 + 리포트 | qa-engineer | 커버리지/통과 게이트 |
| **5. 모니터링** | 모니터링 설계·대시보드·알람 | infra-architect | — |
| **6. 최종점검** | 인수인계 문서(쉽고 명확) | 전 역할 + handover-check | 완성도 체크리스트 |

**흐름**: Claude에게 subagent를 **순서대로 실행** 요청 → 각 단계 결과가 다음 단계 컨텍스트로 전달. `/next-stage`로 단계 진행.

## 5. 핵심 흐름 3가지

### (A) 프로젝트 시작(init) 흐름
```
새 프로젝트 → /init-project
   → project-interview 스킬 발동
   → service-strategist가 AskUserQuestion으로 한국어 심층 인터뷰
   → 답변 기반 scaffold_project.py 실행
   → 루트에 CLAUDE.md + skill.md + docs/ 골격 자동생성
   → 이후 6단계 파이프라인 진행
```

### (B) 필수 리뷰 게이트 강제
```
구현 중 커밋/완료 시점
   → SubagentStop(matcher: frontend-react|backend-docker) hook 발동
   → require_review.py: code-reviewer·security-reviewer 미실행이면 진행 차단(exit 2)
   → 리뷰 통과해야만 다음 단계
※ 위험작업(rm -rf 등)은 PreToolUse hook(block_destructive.py)로 별도 차단
```

### (C) 월 1회 자동 갱신 (당신이 "중요"라 강조)
```
매월 1일 → GitHub Actions(monthly-refresh.yml)
   → deep-research로 최신 트렌드 수집
   → 각 skill의 reference/*.md 갱신
   → plugin.json version bump + 커밋
   → 팀원 Claude Code가 자동 업데이트 감지·적용
⚠️ 세션 예약작업은 7일 후 만료되므로 사용 금지 → GitHub Actions 사용
```

## 6. MCP 활용 (당신 요구: "산출물은 가급적 MCP로")

이 세션에 연결된 MCP를 단계별 산출물 저장·연동에 활용:

| 단계 | 산출물 | 추천 MCP |
|---|---|---|
| 1 인터뷰 | 요구사항·회의록 | Confluence / Notion |
| 2 설계 | 아키텍처·ERD·보안 | Confluence(문서) + Figma(디자인) |
| 3 구현 | 태스크·이슈 | Jira |
| 4 테스트 | 테스트결과·품질 | Confluence + Mixpanel(제품분석) |
| 6 인수인계 | 최종문서 | Confluence(공간 정리) |

> 하나의 skill이 여러 MCP를 조율 가능(검증됨). 단, **어느 MCP를 정식 저장소로 쓸지 §8에서 확정** 필요.

## 7. 설계가 지키는 원칙 (리서치 검증 반영)

1. SKILL.md 본문 **500줄 미만** → 넘치면 `reference/`·`scripts/`로 분리.
2. `description`에 **"무엇을+언제"** 명확히 (자동호출 정확도).
3. code/security-review는 **hook으로 강제**(사람 의지에 안 맡김).
4. 월간 갱신은 **GitHub Actions**(7일 만료 세션task 금지).
5. 역할은 **이름으로 명시 호출**(자동위임 불안정).
6. 각 역할 **최소 도구권한**(보안·집중).
7. 스펙은 **agentskills.io** 추적(옛 레포 스펙 금지).

## 8. ⚠️ 착수 전 확정할 결정사항 (합의 필요)

1. **스킬셋 이름**: `ai-dev-team`(가안) 확정 or 변경?
2. **산출물 정식 저장소**: 로컬 `docs/` / Confluence / Notion / 혼합?
3. **월간 갱신 환경**: 사내 GitHub Actions 가능? (repo 필요) 아니면 로컬/수동?
4. **대상 기술스택 확정**: DB 종류(PostgreSQL? MySQL? Aurora?), 클라우드(AWS 확정?), 백엔드 언어(Node/Python/Java?).
5. **리뷰어 형태**: code/security-review를 **subagent(추천)** or skill로?
6. **1차 제작 범위**: 9개 역할 전부 vs 우선 1~2개(인터뷰+설계) 먼저 파일럿?
